# 班级空间、导航与班级信箱整体重构设计规格

日期：2026-07-10

## 1. 背景

当前公开站点已经具备同学账号登录、人物长廊、公共留言、班级邮局、影像馆、时光轴、年度册和个人主页等能力，也完成了复古纸张视觉与基础响应式改造。

本轮重构不是继续增加彼此孤立的页面，而是根据两张手绘草图重新组织站内信息架构：把公共互动内容集中到新增的“班级空间”，把高频导航收敛为少数清晰入口，把邮局变成随时可达的信箱按钮，并分别为桌面端和手机端设计合适的页面结构。

已确认的产品决策：

- “班级空间”是新增一级页面，不替代“同学档案”。
- 班级空间收纳公共留言板、班级相册预览和班级时间轴预览。
- 登录完成后首先进入前言，即 `/preface`；用户再从前言进入同学档案或其他页面。
- 班级邮局从文字导航中移出，变为导航栏最右侧的信箱图标按钮。
- 影像馆、时光轴和年度册完整页面继续保留。
- 桌面端参考第一张草图的横向导航与班级工作台结构。
- 手机端参考第二张草图的应用式顶部栏、单列空间和独立信箱界面。

## 2. 当前实现核对

### 2.1 已有能力

- `PublicMessageBoard.vue` 已支持公开留言、我的提交、样式选择和审核状态。
- Worker 已支持留言审核、退回原因、精选、置顶和表情反应。
- `MailboxApp.vue` 已支持收件列表和写信。
- Worker 已支持信件详情、已读、回复、管理员信件和系统信件。
- `AlbumGrid.vue`、`timeline.astro` 已具备完整相册和时间轴页面。
- `TopNav.astro` 已具备登录态、未读角标和 Astro 页面切换后的状态同步。
- `ClassmateLoginBook.vue` 已在普通登录和首次改密完成后跳转 `/preface`，符合最新确认的登录落点。

### 2.2 必须在实施中修正的问题

1. 导航把前言、同学录、公共留言、班级邮局、相册、时光轴、年度册全部平铺，桌面端拥挤，移动端层级也不清楚。
2. 导航信箱角标当前读取 `/api/notifications/summary`，没有包含 `/api/mailbox/summary` 的未读信件。
3. `MailboxApp.vue` 尚未接入已有的信件详情、自动标记已读和回复接口。
4. 写信仍要求用户手工填写同学 `slug`，不符合普通用户认知。
5. 留言反应接口已经存在，但公开站点没有展示和操作入口。
6. 相册和时间轴完整接口返回数据较多，不适合在班级空间首屏直接并行拉取完整数据。
7. 当前没有独立账号中心页面，“账号管理”只能依赖个人主页中的零散入口。

## 3. 目标与非目标

### 3.1 目标

1. 新建“班级空间”，作为全班共同内容和互动的公共大厅。
2. 重构全局导航，使结构更像同学录目录页签，同时保留流畅、高级的状态切换。
3. 桌面端和手机端采用不同的内容编排，不再只是同一布局缩窄。
4. 将信件、管理员消息和留言审核通知统一呈现在“班级信箱”体验中。
5. 复用已有业务能力，补齐已经存在但前端未接通的详情、回复、反应和已读功能。
6. 避免新增页面造成首屏请求膨胀、布局跳变或动画重复绑定。

### 3.2 非目标

- 不替换或删除同学档案、影像馆、时光轴和年度册完整页面。
- 不在本阶段重做管理后台整体视觉。
- 不引入重型 3D、全景场景或持续运行的粒子背景。
- 不改变现有同学账号认证方式和 Session Token 机制。
- 不把班级空间做成营销首页、新闻门户或大量卡片堆叠的仪表盘。
- 不在未确认数据需求前引入新的社交关系、关注或即时聊天系统。

## 4. 方案比较

### 4.1 方案 A：只调整导航，保留全部独立页面

将公共留言、相册和时间轴放入“更多功能”，不新增聚合页面。

优点是改动小、风险低。缺点是没有实现草图中的班级公共大厅，页面之间仍然割裂，用户需要反复跳转才能看到班级近况。

### 4.2 方案 B：新增班级空间，保留完整子页面（采用）

班级空间展示留言主舞台、相册精选和时间轴摘要；每个模块保留进入完整页面的入口。导航只保留高频一级入口，信箱独立为图标按钮。

优点是符合草图和已确认需求，能复用已有数据与组件，也给桌面端和手机端留下独立设计空间。缺点是需要增加轻量聚合接口，并对现有留言和邮局组件做边界拆分。

### 4.3 方案 C：将所有功能合并为单页应用工作台

同学档案、留言、相册、时间轴和信箱全部在一个页面内切换。

优点是交互连续。缺点是首屏体积、状态管理和移动端复杂度明显上升，也会削弱同学录“逐页翻阅”的主题，因此不采用。

## 5. 新信息架构

```text
同学录
├─ 首页 /                  未登录封面与登录
├─ 前言 /preface           登录后默认入口与同学录序章
├─ 同学档案 /roster        人物长廊与个人档案入口
│  └─ 个人主页 /student/:slug
├─ 班级空间 /class-space   新增公共大厅
│  ├─ 公共留言板
│  ├─ 班级相册预览 ───────→ /album
│  └─ 班级时间轴预览 ─────→ /timeline
├─ 更多功能
│  ├─ 影像馆 /album
│  ├─ 时光轴 /timeline
│  └─ 年度册 /yearbook
├─ 账号管理 /account       新增轻量账号中心
└─ 班级信箱 /mailbox       右上角图标入口
```

`/messages` 作为已有深链接继续保留，使用与班级空间相同的留言业务组件。它不再占用一级导航位置，后续可以根据访问数据决定是否重定向到 `/class-space#messages`。

## 6. 全局导航设计

### 6.1 桌面端

导航采用“横向纸质目录条”，不是玻璃灯管，也不是普通后台菜单：

- 左侧：书页图标和“同学录”品牌。
- 中部主入口：前言、同学档案、班级空间。
- 中部辅助入口：“更多功能”下拉菜单，收纳影像馆、时光轴和年度册。
- 右侧：账号管理入口和独立信箱图标按钮。
- 信箱图标右上角显示印章红未读数字，超过 99 显示 `99+`。

当前页状态使用“墨迹书签滑片”：一块低饱和纸色底片沿导航项平滑移动，辅以短墨线或压印边缘。动效只使用 `transform` 和 `opacity`，持续时间控制在 220-320ms，不使用发光轨迹和大面积毛玻璃。

信箱按钮必须使用熟悉的信封图标，并提供 `aria-label="班级信箱，N 封未读"` 和悬停提示。未读数字出现或变化时不得改变导航宽度。

### 6.2 手机端

手机端不照搬桌面导航，改为应用式顶部栏：

- 左侧是菜单按钮。
- 中间显示当前空间标题，例如“同学档案”或“班级空间”。
- 右侧固定信箱图标与未读印章。
- 菜单展开为纸页抽屉，包含前言、同学档案、班级空间、更多功能、账号管理和退出登录。
- 当前页面在抽屉中使用书签压印状态，不使用大面积背景高亮。

顶部栏高度固定，标题过长时截断，不允许挤压左右图标。菜单打开时锁定背景滚动，关闭、跳转或 Astro 页面切换后必须释放锁定。

### 6.3 未登录首页

未登录首页继续只展示品牌，不提前暴露完整站内导航。登录状态确认后再显示完整导航，避免首次绘制时菜单先出现后消失。

### 6.4 登录落点

普通登录成功和首次修改密码完成后统一跳转 `/preface`。前言页作为翻开同学录后的序章，继续提供进入同学档案和其他核心页面的清晰入口。

## 7. 班级空间页面

### 7.1 页面定位

班级空间是“全班共同的一张大桌面”：用户进入后先看到大家最近留下的内容，再向下浏览照片和班级事件。它不是三个完整页面的机械拼接，而是三个模块的摘要、操作和继续浏览入口。

### 7.2 桌面布局

页面最大内容宽度建议为 1280px，采用草图中的分区索引和主画布：

1. 左侧窄栏是班级空间目录，包含“公共留言板、班级相册、时间轴”三个锚点。
2. 右侧主区域首先展示大尺寸公共留言舞台。
3. 留言舞台下方紧接输入区、纸张样式选择、字数和发送按钮。
4. 第二段展示 3-4 张相册封面，右侧箭头进入完整影像馆。
5. 第三段展示横向时间刻度和最近 6-8 个事件，末端进入完整时光轴。

左侧目录只在宽桌面端保持粘性；窄桌面和平板改为顶部横向分段目录，避免挤压主内容。

### 7.3 手机布局

手机端按第二张草图重新编排：

1. 顶部栏显示“班级空间”和信箱入口。
2. 首屏是公共留言舞台，只展示精选或最新留言，不在首屏铺满全部卡片。
3. 留言输入采用可展开信纸；未展开时只显示“写一张留言”入口，减少首屏高度。
4. 相册预览使用横向滑动的 2 张半卡片，封面比例稳定。
5. 时间轴使用单侧竖向摘要，不强行保留桌面横轴。
6. 页面内分区切换采用锚点定位，不引入多层横向手势冲突。

所有手机端留言卡默认不旋转，不使用 hover 位移。相册横滑区设置滚动吸附，但不能劫持页面纵向滚动。

### 7.4 公共留言模块

现有 `PublicMessageBoard.vue` 应拆分为可复用的业务边界：

- `MessageComposer`：内容、纸张样式、字数和提交状态。
- `MessageCardGrid`：留言展示、精选/置顶状态和表情反应。
- `MyMessageSubmissions`：当前账号的待审核、已通过和未通过记录。
- `PublicMessageBoard`：独立留言页的编排容器。
- `ClassSpaceMessageStage`：班级空间中的摘要编排容器。

留言提交后只更新留言模块，不重新播放整页入场动画。审核中的留言进入“我的提交”，不能提前混入公共舞台。

表情反应使用已有接口，第一阶段保留四种已有反应，不新增评论回复或无限嵌套讨论。

### 7.5 相册预览

每张预览卡只需要：相册 ID、标题、封面图、照片数量和可选标签。卡片点击进入 `/album` 并定位相应相册，不在班级空间加载灯箱和全部照片。

无封面时使用稳定的纸张占位，不使用随机远程图片。图片必须设置固定 `aspect-ratio`、宽高和懒加载策略，避免加载后跳变。

### 7.6 时间轴预览

桌面端使用横向刻度表达时间流动，手机端切换为单侧竖向摘要。每个节点只展示日期、标题、类型和可选缩略图。完整说明和筛选仍留在 `/timeline`。

## 8. 班级信箱重构

### 8.1 统一内容模型

信箱界面统一展示三类内容，但保留现有数据库边界：

- 管理员或系统通知：来自 `notifications`。
- 同学私信与管理员信件：来自 `mail_threads` 和 `mail_messages`。
- 留言审核结果：继续由审核流程创建通知，在信箱中以“留言消息”分类展示。

前端将它们映射成统一列表项，不强行把通知迁移为邮件线程，避免不必要的数据迁移和重复消息。

信箱页面首次加载时并行请求现有的 `/api/notifications` 和 `/api/mailbox/threads`，在前端映射、合并并按时间排序。第一阶段不新增统一消息列表接口；只有未读摘要需要聚合接口，以兼顾导航性能与现有接口复用。

### 8.2 未读统计

新增统一的信箱摘要响应，返回：

```json
{
  "notificationUnread": 2,
  "mailUnread": 3,
  "totalUnread": 5
}
```

导航只请求一次统一摘要，角标展示 `totalUnread`。打开通知或信件后，当前页和导航角标都应立即更新。

### 8.3 桌面端信箱

桌面端采用列表/详情双栏：

- 左栏顶部提供“全部、管理员、同学私信、留言消息”筛选。
- 左栏列表显示类型印章、发件人、标题、摘要、时间和未读状态。
- 右栏展示完整通知或信件对话。
- 允许回复的私信在线程底部显示信纸回复框。
- 右下角提供写信按钮，打开新信纸编辑面板。

### 8.4 手机端信箱

手机端采用列表和详情两级页面状态：

- 默认只显示消息列表。
- 点击条目进入全屏信纸详情，顶部提供返回按钮。
- 写信按钮固定在安全区上方，不遮挡最后一条消息。
- 返回操作优先恢复信箱列表，不直接离开站点。

### 8.5 写信体验

收件人改为按姓名搜索和选择，不再暴露 `slug`。同学目录数据只在用户打开写信面板时加载。已选择的收件人显示头像、姓名和可移除状态。

标题和正文继续沿用现有限制；不能给自己写信。发送成功后回到对应线程或收件列表，并更新本地状态，不整页刷新。

## 9. 账号管理

新增 `/account` 轻量账号中心，解决导航中“账号管理”没有实际落点的问题。第一阶段只包含：

- 当前账号姓名、头像和账号状态。
- 进入自己的个人主页。
- 打开个人资料自助编辑。
- 修改登录密码。
- 退出登录。

账号中心不复制个人主页全部资料，也不承载管理员功能。它只负责账号级操作和跳转。

## 10. 数据与接口设计

### 10.1 班级空间摘要接口

新增 `GET /api/class-space/overview`，只返回班级空间所需的轻量数据：

```json
{
  "messages": [],
  "albums": [],
  "timeline": [],
  "counts": {
    "approvedMessages": 0,
    "albums": 0,
    "timelineItems": 0
  }
}
```

约束：

- 留言最多返回 8 条，只包含已审核内容。
- 相册最多返回 4 条，每条只返回封面和计数，不返回完整照片数组。
- 时间轴最多返回 8 条，只返回摘要字段。
- 不返回同学私密资料、Session 信息或邮箱数据。
- 响应允许短时缓存和 `stale-while-revalidate`，减少 D1 重复查询。

班级空间打开后只需要一次摘要请求。用户展开“我的提交”、写信或进入完整页面时，再按需请求对应接口。

### 10.2 统一信箱摘要接口

新增 `GET /api/inbox/summary`，需要同学 Session，聚合通知和邮件未读数。现有 `/api/notifications/summary` 与 `/api/mailbox/summary` 暂时保留，避免破坏旧客户端和测试。

### 10.3 前端状态边界

- `ClassSpaceHub` 只负责班级空间摘要加载和三个模块之间的共享状态。
- 留言提交、表情反应和我的提交由留言业务模块管理。
- `MailboxApp` 负责筛选、列表、详情、回复和写信状态。
- 导航只关心登录身份和统一未读数，不持有完整信箱数据。
- Astro 页面切换后重新同步必要状态，但不得重复注册监听器或轮询器。

## 11. 动画与交互规范

### 11.1 导航动画

- 当前项滑片使用 `transform`，不通过改变宽高推动布局。
- 下拉菜单使用 160-220ms 的淡入和短距离位移。
- 手机抽屉使用一次进入/退出动画，不叠加全局 reveal。
- 信箱角标只做轻微缩放反馈，不持续闪烁。

### 11.2 班级空间动画

- 页面首次进入时，三个分区按阅读顺序渐显一次。
- 留言卡采用轻微“放上桌面”的位移和透明度变化；手机端取消旋转。
- 相册横滑依赖原生滚动，不使用持续拖拽动画库。
- 时间轴只在首次进入视口时显示节点，不因筛选、提交留言或页面回退重复播放整段动画。

### 11.3 生命周期与降级

- 一个元素只能由全局 reveal 或组件局部动画中的一个系统控制。
- 所有监听器、Observer 和定时器在 Astro 页面切换或组件卸载时清理。
- 动画只操作 `transform` 和 `opacity`，不持续动画化阴影、模糊和布局属性。
- `prefers-reduced-motion` 下取消滑片追随、错落入场和滚动吸附动画，保留即时状态变化。

## 12. 加载与稳定性

- 班级空间先绘制稳定的纸张骨架，再填充摘要数据，骨架高度与最终模块接近。
- 首屏只加载留言舞台和必要导航脚本；相册图片和时间轴缩略图在接近视口时加载。
- 不在班级空间加载完整 `AlbumGrid`、完整时间轴渲染器、灯箱、GSAP 或 ScrollTrigger。
- 摘要加载失败时，各模块分别显示纸张错误状态和重试按钮，不能让整页空白。
- 留言提交失败时保留用户输入。
- 信箱列表加载失败、详情加载失败和回复失败分别处理，不清空已经加载的数据。
- 所有图片和异步区域预留稳定尺寸，避免重复加载和布局跳变。

## 13. 可访问性与隐私

- 导航、菜单、信箱筛选和留言筛选使用正确的 `aria-current`、`aria-expanded` 和可见焦点。
- 所有图标按钮都有中文可访问名称和工具提示。
- 手机触控目标不小于 44px。
- 信件详情、通知和收件人目录只能在有效同学 Session 下获取。
- 后端继续校验信件创建者和收件人身份，前端隐藏入口不能替代权限校验。
- 留言只展示已通过审核的公共内容，“我的提交”只对提交者本人可见。

## 16. 风险与控制

- **范围扩张：** 班级空间、导航、信箱和账号中心必须按阶段交付，不同时重写无关个人主页内容。
- **接口重复：** 新聚合接口只做读取与裁剪，不复制留言、相册和时间轴写入逻辑。
- **消息重复：** 通知和信件在 UI 中统一展示，但数据库继续分离；同一审核事件只产生一条通知。
- **导航闪烁：** 登录态和未读数使用稳定占位，角标变化不能推动布局。
- **移动端手势冲突：** 页面纵向滚动优先，相册横滑区限制在明确区域，留言板不使用自由拖拽。
- **动画回归：** 所有局部动效必须声明所有权并支持清理和 reduced motion。
- **旧链接失效：** 现有页面和深链接在本阶段保留，不直接删除路由。

6. 账号管理入口有真实页面和可用操作。
7. 没有重复监听、重复入场动画、明显布局跳变或横向溢出。
8. 自动测试、手动视觉 QA 和发布前验证全部通过。
