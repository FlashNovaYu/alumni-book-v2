# 同学档案身份共享转场设计

日期：2026-07-12

## 背景

当前“人物长廊”卡片进入标准学生详情页时，整个卡片与详情页根节点共用 `active-card`。点击后卡片中的头像和正文会提前淡出，再由详情页整体淡入。因此视觉上会出现“姓名和头像先消失、再出现”，无法形成用户希望的连续移动感。

用户希望借鉴 Dan's Blog 在 Vlog 和 Tags 等栏目切换时的文字连续换写：同一位同学的头像和姓名应被视为跨页面的同一视觉实体。进入详情时，它们从卡片位置移动、缩放到个人页 Hero；返回人物长廊时反向回到原卡片。

## 目标与边界

### 目标

1. 从可访问的学生卡片进入其标准个人页时，头像由卡片中的 72px 圆形连续移动、缩放到 Hero 中的 96px 圆形。
2. 同一学生的姓名从卡片正文位置连续移动、缩放到个人页 `h1`，不中途淡出或重新出现。
3. 个人资料的其余内容在头像和姓名开始落位后再短暂淡入，保持层次而不遮挡共享元素。
4. 浏览器返回人物长廊时自动反向播放相同的身份连续性。
5. 减少动态偏好或不支持 View Transitions 时，内容仍立即、完整地可见并正常跳转。

### 非目标

- 不改变学生 API、资料字段、路由、鉴权或专属页面模板。
- 不给每张卡片添加手写 FLIP 副本、固定定位克隆节点或额外动画库。
- 不改变已经独立工作的信箱水波 `vt-mailbox`、主题水波 `theme-transition` 和一级栏目标题 `page-heading`。
- 不让自定义 `isOwner + customHtml` iframe 模板错误继承标准个人页的身份转场。

## 已比较方案

### 方案 A：头像和姓名各自作为共享元素（采用）

两端使用由学生 `slug` 派生的稳定名称：`student-avatar-<slug>` 与 `student-name-<slug>`。浏览器的跨文档 View Transition 负责几何插值；卡片底板和非身份文字不再作为同一共享元素。这与 Dan's Blog 使用稳定共享名称的模式一致，同时通过 `slug` 避免列表中多张卡片发生名称冲突。

优点是使用现有 Astro `ClientRouter` 和原生 View Transitions，DOM 最少、支持浏览器后退、不会产生手写层的滚动或焦点同步问题。缺点是只在支持该 API 的浏览器中产生连续形变，但已有普通导航回退可安全覆盖其他环境。

### 方案 B：保留整卡片共享元素，同时叠加头像和姓名

这会使父元素和子元素在同一次转场中被同时截图，容易出现双重快照、边框闪烁、文字鬼影；不采用。

### 方案 C：手写 FLIP 临时副本

可以更精确地控制每一帧，但需要处理路由交换、图片加载、滚动、返回、焦点和减少动态等状态，远超本需求；不采用。

## 设计

### 共享命名和组件职责

`ArchiveRosterCard.vue` 只负责在一次会跳转的卡片点击后，为自身头像与姓名写入对应的 View Transition 名称：

- 头像容器：`student-avatar-${card.slug}`
- 姓名元素：`student-name-${card.slug}`

`StudentProfile.vue` 在加载到标准模板学生资料后，使用同一名学生 `student.slug` 为 Hero 头像和 `h1` 写入相同名称。名称仅在该组件对应的学生数据就绪时存在。头像图片失败时，首字母占位符仍置于同一头像容器内，因此转场不会丢失。

非可访问卡片（`hasPage === false`）不写共享名称，也不触发转场状态。专属 iframe 模板没有标准 Hero 目标，不能写身份共享名称。

### 时间与视觉层次

1. 点击卡片后，卡片保持可见，只有头像与姓名成为共享元素；卡片底板、格言、标签与状态不再等待或抢占共享截图。
2. 共享头像在约 460–520ms 内采用现有阻尼缓动移动并从 72px 缩放至 96px；边框和圆角由浏览器快照连续插值。
3. 共享姓名在约 360–420ms 内从卡片正文移动至 Hero `h1`，使用轻微透明度和去模糊，但不得先被设为 `opacity: 0`。
4. Hero 中昵称、展柜说明、资料完整度、格言、标签和操作按钮，以及下方资料正文，在共享身份元素开始稳定后（约 160–220ms）向上 12–16px 淡入。共享头像与姓名不得包含在此延迟淡入容器中。
5. 返回操作利用同名元素和浏览器方向自动反向；不为返回单独维护状态。

全程保持纸张、档案和印章语言：不新增玻璃面板、3D 翻页、循环粒子或弹跳效果。

### 现有 `active-card` 的迁移

该路径不再在卡片根节点或标准学生页根节点使用 `active-card`，并删除点击时将卡片头像/正文设为 `vt-fade-out` 的行为和对应样式。`active-card` 的全局规则可以保留给其他未来使用方；若当前没有任何使用方，应在本次改动中仅删除与该路径绑定的规则，避免留下会误导维护者的死功能。

全局样式新增仅针对 `student-avatar-*` 和 `student-name-*` 的 View Transition 伪元素规则。头像和姓名各有独立时长与相同的 `cubic-bezier(0.16, 1, 0.3, 1)`，以免卡片或页面根级 `page-main` 动画覆盖其快照层。详情非身份内容的延迟淡入应限定到新的语义类名，而不是 `.hero-content` 整体。

### 降级、可访问性与性能

- `prefers-reduced-motion: reduce` 下，移除所有共享名称和详情延迟动画；点击后正常跳转，头像、姓名、正文均保持可见。
- 没有 `document.startViewTransition` 的浏览器会走 Astro 的普通页面导航；不得依赖 JavaScript 克隆或隐藏内容作为前提。
- 姓名文本保持真实 DOM 文本；图片的 `alt` 文本和链接键盘可达性不变。
- 动画只使用浏览器快照与 `transform`/`opacity`/`filter`，不绑定滚动、鼠标移动或持续计时器。

## 测试与验收

静态 Vitest 回归应确认：

1. 卡片和标准详情页都由同一 `slug` 生成头像、姓名两个不同的共享名称。
2. 卡片不再声明 `active-card` 或在该路径使用 `vt-fade-out`；详情页根节点同样不声明 `active-card`。
3. 非身份详情内容使用独立延迟类；共享头像与姓名不在该类中。
4. CSS 为两种身份名称提供独立转场规则，并且减少动态偏好下禁用共享名称和详情延迟动画。
5. 专属 iframe 分支不提供身份共享目标。

Playwright 回归应确认：

1. 打开人物长廊，点击一个有个人页的卡片后，目标页面的 Hero 头像和姓名使用与点击卡片相同的 `view-transition-name`。
2. 使用浏览器后退回到人物长廊后，对应卡片仍可见且其头像、姓名共享名称完整恢复。
3. 减少动态上下文中点击卡片不会让头像、姓名、正文出现 `opacity: 0` 或遗留转场状态。
4. 既有夜读水波、信箱水波和一级栏目标题测试继续通过。

人工验收在桌面与 390px 移动宽度完成：确认头像和姓名看起来是“移动到个人页”，而不是消失后重新渲染；卡片的其余信息自然退场，详情其余资料稍后进入；返回方向同样连贯。
