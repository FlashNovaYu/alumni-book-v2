# 后台网络层加固设计

## 背景

线上后台已经通过构建环境变量使用 Pages 同源 `/api`，但源码默认值仍指向 `workers.dev`，登录、路由鉴权和普通接口各自直接调用 `fetch`，缺少一致的超时、重试和错误语义。后续误用默认构建命令时，仍可能把跨域 Worker 地址重新打入产物。

本次只优化后台网络层，不修改任何 Vue 模板、样式、字体或视觉资源，也不改 API 业务协议。

## 目标

- 后台生产构建默认使用同源 `/api`，仅在开发者显式设置 `VITE_API_BASE_URL` 时连接其他后端。
- 普通 JSON 请求、登录和 Token 校验共用一套请求实现。
- 普通请求 15 秒超时，上传请求 60 秒超时。
- 仅 GET 请求在网络异常或 502、503、504 时自动重试一次。
- POST、PUT、DELETE 和上传请求绝不自动重试，避免重复写入。
- 401 行为保持不变：普通后台请求清除 Token 并进入登录页；登录失败只显示后端错误，不触发额外跳转。
- 构建产物出现 `workers.dev` 或其他绝对 API 地址时，质量门禁失败。
- 先验证非生产预览，不直接更新生产部署。

## 方案

新增一个与 Vue 无关的网络模块，负责 URL 拼接、超时、幂等重试、响应解析和统一错误。模块返回明确的 `ApiRequestError`，携带 HTTP 状态码，认证客户端据此决定是否清除会话。

`client.ts` 只负责后台 Token、401 跳转以及登录/退出等认证语义；路由守卫通过该客户端验证 Token，不再维护第三套直接 `fetch` 逻辑。现有各页面继续调用 `adminFetch`，因此页面组件无需改动。

同源策略在 Vite 配置中固化为空字符串默认值。新增构建产物扫描脚本并接入 `verify:admin`，从发布流程上阻止跨域 API 地址回归。

## 错误与重试规则

| 情况 | 对外结果 | 是否重试 |
| --- | --- | --- |
| GET 网络异常 | `网络连接失败，请检查网络后重试` | 一次 |
| GET 返回 502/503/504 | 最终返回服务暂不可用错误 | 一次 |
| 请求超时 | `请求超时，请稍后重试` | GET 一次，其他方法不重试 |
| 4xx | 使用后端 `message`，否则使用状态码 | 否 |
| 写操作网络异常 | 网络错误 | 否 |
| JSON 响应格式异常 | `服务器响应格式异常` | 否 |

## 测试策略

- 使用 Node 测试运行器配合 `tsx` 测试纯网络模块，覆盖 URL、GET 重试、写操作不重试、超时、后端错误和无效 JSON。
- 使用静态契约测试确认认证客户端和路由守卫统一走网络模块。
- 构建后台后扫描全部 HTML、JS 和 CSS 产物，确认不存在 `workers.dev` 或绝对 API 基址。
- 运行后台测试、类型检查、生产构建，再运行全仓库质量门禁。
- 部署 Pages 非生产预览，检查新界面资源、同源 API、登录失败路径以及浏览器网络请求。

## 非目标

- 不移除 Google Fonts，不调整字体或任何视觉细节。
- 不引入 Service Worker、离线缓存或后台数据缓存。
- 不自动重试写请求。
- 不修改 Worker API、D1 或 R2 数据结构。
