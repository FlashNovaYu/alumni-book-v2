# 同学档案卡双向边缘转场设计规格

## 目标

恢复同学档案卡进入个人页时的连续转场，并升级为“双向边缘”效果：不再叠加整页左右退出/进入；被点击卡片的下一页底色从卡片矩形四边向外扩散；卡片中除头像、姓名外的信息向内坍缩；头像与姓名从原卡片被保留为共享身份元素，在扩散期间同步缓慢移动，覆盖完成后落入个人页 Hero。

## 当前根因

当前自托管线上版本与本地提交一致，动效缺失来自稳定性修复而非发布遗漏：

- `packages/site-astro/src/styles/view-transitions.css` 将导航设置为 `navigation: none`，并用全局规则将 `view-transition-name` 设为 `none`。
- `packages/site-astro/src/layouts/MainLayout.astro` 只保留普通链接导航、顶部进度线和会话状态恢复，不再让 Astro ClientRouter 接管页面交换。
- `ArchiveRosterCard.vue` 与 `StudentProfile.vue` 仍可生成身份 transition name，但上述 CSS 会将其禁用。

## 交互时序

1. 用户点击拥有标准个人页的档案卡。
2. 在旧页面的 `astro:before-preparation`/点击捕获阶段读取卡片矩形、头像矩形和姓名矩形；记录导航方向只用于无动画退化和状态清理，不驱动整页左右位移。
3. 为下一页准备一个固定层：背景色使用下一页主题对应的纸页/夜读背景；起始矩形与被点击卡片边界完全重合，零圆角、零缝隙。
4. 覆盖层四边同时向外扩展；旧页面保持为转场快照底层，不再做整页向左/右退出，避免与卡片边缘扩散叠加出两套方向运动。
5. 卡片的座右铭、标签、学校等非身份信息在同一时间从四边向中心坍缩并淡出，不参与跨页共享。
6. 头像与姓名被提升到转场前景层，覆盖层扩散期间从原卡片位置向个人页 Hero 位置缓慢移动；移动路径不能先于覆盖边界进入未覆盖区域。
7. 覆盖层达到全屏后，头像和姓名完成最后一段对位，个人页辅助内容再按现有 `detail-content-enter` 进入。
8. 动画结束后清理所有临时层、sessionStorage 状态、方向标记和 transition name，避免影响下一次导航。

## 方案与取舍

采用 Astro ClientRouter + View Transition API 的受控跨文档转场，原因是它能在同一浏览上下文中同时获得旧文档、新文档和共享元素快照，避免依赖“旧页面等待网络后再跳转”的脆弱时序。实现只恢复必要的导航接管，并将整页 root 的默认左右退出/进入设为静止，让卡片边缘扩散成为唯一主转场。

具体约束：

- 根页面层只负责承载新旧快照和矩形底色扩散；共享头像、姓名层拥有更高的 transition group 层级。
- 非身份卡片信息不设置共享 `view-transition-name`，通过旧页面卡片内容的 `scale/opacity` 规则完成坍缩。
- 不使用全局 `html [style*="view-transition-name"] { view-transition-name: none !important; }` 这类会误杀身份层的兜底规则；只在 `prefers-reduced-motion` 下关闭。
- 不把圆形水波用于档案卡转场；明暗模式的圆形水波保持现有实现。
- 不改变 API、数据结构、身份认证或自托管构建参数。

## 兼容与退化

- 支持 View Transition API 且用户未启用减少动态效果时，执行完整双向边缘转场。
- 不支持 View Transition API、用户启用减少动态效果或转场准备异常时，保留现有普通导航，并确保页面立即可用；不得阻塞链接跳转。
- 转场颜色从当前主题 CSS 变量读取，不写死 Cloudflare 或远程 API 地址，阿里云自托管构建保持同源 API。

## 验收标准

- 点击档案卡时不再出现整页左右滑动，主要运动来自卡片四边同步外扩与卡片内容内坍缩。
- 覆盖层起点与卡片矩形完全重合，四边同时扩散，过程中无圆角空隙或白色闪烁。
- 头像和姓名在转场期间始终可见、从原卡片位置连续移动，且不早于覆盖边界移动到未覆盖区域。
- 座右铭、标签、学校等其他卡片信息向内坍缩并消失，不会与头像或姓名重复出现。
- 覆盖完成后头像和姓名准确落到个人页 Hero，个人页辅助内容正常进入。
- 前进与后退导航方向均成立；共享元素不存在时仍可正常导航。
- `prefers-reduced-motion: reduce` 下不执行转场动画，不影响可用性。
- 自托管构建产物不包含 Cloudflare 地址，线上 `/release.json` 仍指向实际发布提交。
