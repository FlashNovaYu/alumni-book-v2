# 温润青春档案交互音效设计说明书

## 1. 目标

为公开站点建立一套统一、克制且具有触感的交互音效，强化“青春纪念馆 + 互动手账 + 纸质档案”的主题。音效应让用户感到自己正在翻阅一册被认真保存的同学录，而不是操作一个播放提示音的后台系统。

本设计只覆盖公开站点的短交互音效，不改变同学个人页可配置的背景音乐，也不改变现有静音开关、用户偏好持久化或首次用户手势后才创建 `AudioContext` 的行为。

## 2. 现状与问题

当前音效运行时位于 `packages/site-astro/src/runtime/audioSynth.ts`，主要提供 `playCrystalTick`、`playPaperSlide` 和 `playDeepWhoosh`。现有触发关系如下：

- 全局导航和音量按钮悬停复用短促 click。
- 同学卡片、照片卡片悬停或触摸复用纸张滑动声。
- 相册打开又复用短促 click。
- `playDeepWhoosh` 目前没有实际的公开站点触发点。

现有实现存在四个直接影响质感的问题：

1. 声音名称、材质和交互语义不一致，所有短确认声听起来像同一个泛用 beep。
2. 纸张噪声缓冲只有约 100ms，却被安排了更长的衰减时间，尾部会突然截断。
3. 各声音直接连接到 `destination`，没有统一音频总线、轻量压缩或并发声部控制，连续悬停容易叠音或显得过响。
4. 高频噪声和低频下坠都过于直白，缺少“近距离、温润、被纸张包住”的听感。

## 3. 声音语言

### 3.1 材质

- 纸张：柔和的中频摩擦，避免持续白噪声和刺耳高频。
- 铅笔：短、干、近距离的木质触点，不做金属键盘感。
- 胶片相机：轻微机械遮罩感，加极短的空气释放，不模拟夸张快门。
- 木桌与书页：只保留低中频的温暖支撑，不制造影院式 sub-bass。

### 3.2 混音原则

- 所有 UI 音效保持干声为主，仅使用极短、低比例的空间尾音。
- 采用轻微随机化（音高、起始相位或噪声片段），避免连续触发完全相同。
- 单次音效目标时长为 30–300ms，禁止拖成长背景音。
- 低于 80Hz 的内容只作为极低比例的触感，不承担主要辨识度。
- 声音峰值与背景音乐共存时不抢人声；UI 音效必须有统一总线音量和限幅。

## 4. 音效角色与参数目标

以下是实现验收时的听感目标，不要求逐样本复刻固定波形。

| 角色 | 语义与层次 | 目标时长 | 目标触发 |
| --- | --- | ---: | --- |
| `archiveHover` 铅笔轻点 | 木质微击 + 很短的柔和高频尾音；音高轻微随机化 | 30–45ms | 导航、按钮和可进入卡片的首次悬停 |
| `archiveSlide` 照片纸移位 | 低比例纸张摩擦 + 600Hz 至 1.7kHz 的柔和滤波扫过 | 120–160ms | 同学卡片、照片卡片悬停或触摸 |
| `albumOpen` 快门取景 | 轻机械遮罩 transient + 极短上行双音，像胶片相机完成取景 | 160–220ms | 打开相册灯箱或照片详情 |
| `bookSettle` 书页落桌 | 温暖低中频支撑 + 纸页收束；不做轰鸣或明显下坠 | 220–300ms | 抽屉关闭、页面切换或档案视图收束 |
| `archiveConfirm` 邮戳回执 | 闷纸章触点 + 短促上行双音，表达“已认真收下” | 180–230ms | 开启音效、保存或发送成功 |

## 5. 现有触发点映射

第一阶段只修改已有公开站点交互，不扩展后端或数据库：

- `volumeToggle.ts` 的导航悬停从 `playCrystalTick` 改为 `archiveHover`。
- `ArchiveRosterCard.vue` 的悬停/触摸从 `playPaperSlide` 改为 `archiveSlide`。
- `AlbumGrid.vue` 的悬停/触摸使用 `archiveSlide`，打开灯箱改用 `albumOpen`。
- 音量从静音切换为开启时使用 `archiveConfirm`，不再复用悬停 click。
- `playDeepWhoosh` 改名并收敛为 `bookSettle`；仅在确认已有页面/抽屉状态事件后接入，不能为了“用上音效”新增无语义触发。

保存/发送成功音效先保留运行时能力和明确命名；如果当前公开站点没有稳定且不重复的成功事件，第一阶段只接入音量开启，后续在对应功能验收时再接入保存/发送。

## 6. 技术方案

采用“少量质感微采样 + Web Audio 程序化层”的混合方案：

1. 木质触点、纸张摩擦和相机遮罩使用极短微采样，首次允许播放的用户手势触发音效后惰性加载；不在页面加载阶段请求。
2. 音高、衰减、滤波扫频和轻微随机化继续由 Web Audio 完成，使一个素材可以覆盖多个相近变体。
3. 微采样加载失败时，必须回退到现有运行时内的程序化合成，不得让交互抛出未捕获异常。
4. 增加共享 master bus：统一音量、轻量 compressor 和安全限幅；各声部不再直接连接 `destination`。
5. 增加短时 voice budget：同一时间最多保留两个 UI 音效声部；新的同类声音应优先截断旧的悬停声，而不是无限叠加。
6. 保留现有静音状态的 `localStorage` key、页面隐藏时暂停/恢复和首次用户手势创建 `AudioContext` 的性能约束。
7. 悬停触发继续节流，目标窗口约为 100–120ms；触摸事件不得与紧随其后的点击重复播放同一语义音。

## 7. 性能、可访问性与兼容性

- 默认不预加载背景音乐或 UI 音效素材；音频资源只在首次允许播放的用户手势触发音效后请求。
- 微采样总传输预算目标不超过 35KB（压缩后）；超过预算必须说明原因并重新评估。
- 不使用 `ScriptProcessorNode`，不创建常驻定时器，不在页面加载阶段创建 `AudioContext`。
- 静音按钮继续提供中文 `aria-label`、`aria-pressed` 和 `title`，状态与实际音频状态一致。
- 浏览器不支持 Web Audio 或素材加载失败时，所有页面功能仍可正常使用。
- `prefers-reduced-motion` 不应被误解为强制静音；是否播放由用户的音效开关决定。

## 8. 验收标准

### 听感验收

- 导航悬停不再像尖锐金属 click；连续移动鼠标时有轻微变化但不杂乱。
- 卡片悬停听起来像纸面或照片被轻轻移开，不出现明显的白噪声尾部截断。
- 打开照片时能听出“取景/快门”语义，但不会盖过页面背景音乐。
- 页面或抽屉收束时有温暖的落桌感，不出现影院低频轰鸣。
- 开启音效时有独立的确认语义，和悬停音可以一耳朵区分。

### 工程验收

- 现有静音持久化、音频上下文惰性创建、页面隐藏暂停/恢复测试继续通过。
- 新增静态测试覆盖：声音角色导出、所有声部连接 master bus、并发预算、素材失败 fallback 和触发点映射。
- 浏览器测试确认：未主动开启音效时不创建 `AudioContext`，首次开启后只创建一个上下文；连续悬停不会无限增长 AudioBufferSourceNode。
- `pnpm build:site`、站点相关单元测试和浏览器测试通过。
- 通过桌面 Chrome、移动 Chrome 至少各试听一次，并在没有背景音乐时确认音效不刺耳；不输出或提交任何密钥、用户数据或临时录音文件。

## 9. 非目标

- 不重做学生可配置背景音乐播放器。
- 不引入全站持续环境音、自动播放音乐或需要用户授权的麦克风能力。
- 不把每个页面都强行接入声音；只有有明确动作语义的交互才播放。
- 不修改当前用户未提交的业务代码、认证流程、API、数据库和部署链路。
