# 班级空间、群聊、信箱与导航返工设计规格

日期：2026-07-10

## 1. 背景与返工原因

上一阶段已经新增班级空间、信箱入口和账号中心，但人工验收仍未达到设计目标：

1. 班级空间仍像三个普通模块的纵向拼接，没有忠实呈现手绘草图中的左侧目录、群聊主舞台、横向相册和桌面时间刻度。
2. 公共留言仍是只读留言卡片，没有群聊需要的即时发送、引用、回应、撤回和增量更新。
3. 班级信箱仍按“主题邮件”组织。同一位同学会产生多个线程，且现有数据库只登记初始收件人，导致发件人列表缺失、回复未读无法正确回流。
4. 导航仍是固定宽度的悬浮玻璃胶囊，与复古纸张界面割裂；手机端缺少中间页面标题和固定信箱入口。
5. 多处链接硬编码为根路径，未完整遵守 `SITE_BASE`；部分新增 Playwright 文件也没有进入正式质量门禁。

本规格是在保留现有账号、前言、同学档案、相册、时间轴和通知能力的前提下，对上述三个核心体验做一次完整返工。它取代原规格中关于“公共留言卡片”“主题邮件信箱”和“悬浮胶囊导航”的相关设计。

## 2. 已确认的产品决策

### 2.1 信息架构

- 登录和首次修改密码完成后统一进入 `/preface`。
- `/roster` 继续作为同学档案入口。
- `/class-space` 是一级页面，内部承载公共群聊、班级相册摘要和班级时间轴摘要。
- `/messages` 不再维护独立公共交流界面，统一转到 `/class-space#group-chat`。
- 班级信箱通过导航右侧信箱按钮进入，不占用一级文字导航。
- `/account` 继续承载个人资料、密码和退出登录。

### 2.2 公共群聊

- 新消息采用“先发后审”：合法消息发送成功后立即对全班可见。
- 第一阶段支持文字、Emoji、引用回复、四种表情回应、作者撤回和管理员撤回。
- 管理员可以隐藏消息、撤回消息、禁言账号，并向当事人发送处理通知。
- 图片继续进入班级相册，不在群聊内发送。
- 已通过审核的历史公共留言进入群聊历史。
- 历史待审核和未通过留言不进入群聊，只保留在本人记录和管理后台。

### 2.3 班级信箱

- 每两位同学之间只有一个持续会话，不再按标题创建多个邮件线程。
- 私聊与通知分开。
- 管理员消息、系统消息和审核处理结果进入“通知”。
- 私聊支持自动增量刷新、未读数、自动滚动、发送状态和失败重试。
- 第一阶段不做在线状态、正在输入、对外可见的已读回执、图片和文件发送。

### 2.4 视觉方向

- 桌面导航采用全宽纸质档案目录条，完整移除玻璃发光胶囊。
- 桌面班级空间采用手绘草图中的“班级工作台”。
- 手机班级空间采用“连续分段长页”，不是桌面布局缩窄版，也不是三个孤立标签页。
- 继续保留流畅动画，但动画必须服务于页面层级和状态反馈，不持续发光或循环播放。

## 3. 目标与非目标

### 3.1 目标

1. 让班级空间成为可真实使用的公共群聊大厅，并保持复古纪念册的视觉气质。
2. 让任意两位同学拥有唯一、持续、双向可见的私聊会话。
3. 让管理员通知、系统通知和审核结果形成独立、清晰的通知流。
4. 让桌面端和手机端分别拥有适合自身操作方式的布局。
5. 消除导航和聊天的重复初始化、重复请求、动画重放与布局跳变。
6. 完整保留旧数据，并通过可验证迁移修复现有邮件参与者关系。

### 3.2 非目标

- 不引入 WebSocket、Durable Objects 或第三方即时通讯服务；第一阶段使用可控的增量轮询。
- 不支持群聊图片、语音、文件、消息编辑、无限嵌套回复或群成员在线状态。
- 不支持私聊主题、群聊创建、好友关系、拉黑和多成员私聊。
- 不重做整个管理后台，只改造与群聊审核、禁言和通知发送直接相关的页面。
- 不删除个人主页留言、完整相册页、完整时间轴页和年度册。
- 不引入 GSAP、持续粒子、3D 场景或重型动画依赖。

## 4. 最终信息架构与用户路径

```text
登录 / 首次改密
└─ 前言 /preface
   ├─ 同学档案 /roster
   │  └─ 个人主页 /student/:slug
   ├─ 班级空间 /class-space
   │  ├─ 公共群聊 #group-chat
   │  ├─ 班级相册摘要 #albums ─────→ /album
   │  └─ 班级时间轴 #timeline ─────→ /timeline
   └─ 更多功能
      ├─ 影像馆 /album
      ├─ 时光轴 /timeline
      └─ 年度册 /yearbook

任意站内页面
├─ 信箱按钮 /mailbox
│  ├─ 私聊
│  └─ 通知
└─ 账号中心 /account
```

路由规则：

- `/messages` 使用站点基础路径生成 302/静态重定向，目标为 `/class-space#group-chat`。
- 旧书签和外部链接继续可用，但不再下载旧公共留言页面组件。
- 所有导航和组件链接统一通过项目现有的基础路径 helper 生成，禁止硬编码 `/album`、`/timeline`、`/messages` 等根路径。

## 5. 全局导航设计

### 5.1 桌面端

导航是页面顶部全宽、固定高度的纸质档案目录条：

- 左侧：书册标记和“青春纪念册”品牌。
- 中间：前言、同学档案、班级空间、更多功能。
- “更多功能”收纳影像馆、时光轴和年度册。
- 右侧：账号入口和信箱图标，信箱显示固定尺寸的印章红未读角标。
- 导航使用 `position: sticky`，高度始终为 64px；滚动后只增加轻微底部分隔和阴影，不改变高度、宽度或上下位置。

当前项采用“淡色档案签 + 底部墨线”：

- 首次渲染直接处于正确位置，不从页面左侧飞入。
- 站内切换时，墨线以 `transform` 和 `width` 在旧项与新项之间过渡，时长 240ms。
- 动画结束后清除临时内联变换，避免窗口缩放后位置错误。
- 不使用无限循环的墨线呼吸动画，不使用 `backdrop-filter` 和发光阴影。

“更多功能”必须同时支持点击、键盘和焦点进入，不能只依赖 `:hover`。点击外部、按 `Escape`、路由切换和失去有效焦点时都要关闭。

### 5.2 手机端

手机顶部栏固定为三栏、52px 高：

- 左侧为菜单图标按钮。
- 中间显示当前页面标题，例如“前言”“同学档案”“班级空间”。
- 右侧为信箱图标与未读角标。

中间标题使用 `min-width: 0`、单行省略和固定左右图标轨道，不能挤压按钮。菜单从左侧打开为纸页目录抽屉，包含前言、同学档案、班级空间、更多功能、账号中心和退出登录。

抽屉打开时锁定页面滚动；关闭、跳转、`astro:before-swap` 和组件销毁时都必须恢复滚动。手机顶部栏不使用桌面品牌文字替代当前页面标题。

### 5.3 生命周期所有权

导航运行时只能有一个所有者：

- 全局监听器由单一模块注册一次，并保存在 `window` 级运行时对象中。
- `astro:page-load` 只重新取得新 DOM 引用并同步状态，不重复注册 `scroll`、`resize`、点击和未读轮询。
- `astro:before-swap` 中关闭菜单、取消在途请求并释放旧 DOM 引用。
- 信箱摘要请求同一时刻最多一个；新请求开始前取消旧请求。
- 站内 `alumni:inbox-changed` 事件触发即时同步，常规未读轮询仅在登录且页面可见时每 60 秒执行一次。

## 6. 班级空间布局

### 6.1 桌面端：班级工作台

内容最大宽度为 1280px，主结构为 `176px + minmax(0, 1fr)`：

1. 左侧纸质目录包含“公共群聊、班级相册、班级时间轴”三个锚点。
2. 右侧首先是大尺寸群聊舞台，可视高度设为 `clamp(520px, 68vh, 720px)`。
3. 群聊输入栏固定在舞台内部底部，不随外部页面滚动离开聊天区域。
4. 群聊下方是横向相册轨道，桌面一次展示 3-4 张照片封面。
5. 最下方是横向班级时间刻度，节点按日期从左到右排列。

左侧目录在宽度不小于 1100px 时粘性定位；在 769-1099px 之间改为内容顶部横向锚点条。锚点高亮由 `IntersectionObserver` 驱动，但只创建一个 Observer。

### 6.2 手机端：连续分段长页

手机端不保留左侧栏，采用以下顺序：

1. 顶部栏下方是横向分区锚点：群聊、相册、时间轴。
2. 群聊占据首个主要视区，聊天记录区高度使用 `clamp(420px, 62svh, 620px)`，内部独立滚动。
3. 输入栏粘在群聊容器底部，并为系统安全区预留 `env(safe-area-inset-bottom)`。
4. 群聊之后自然进入横滑相册，一次露出约 1.4-2.2 张照片，提示仍可继续滑动。
5. 最后是横向可滑动的时间刻度，不改回纵向卡片时间轴。

相册和时间轴使用各自的横向滚动容器与 `scroll-snap`，但必须设置正确的 `touch-action`，不能阻断页面纵向滚动。手机消息气泡不旋转，任何模块都不能产生页面级横向溢出。

### 6.3 页面结构边界

组件职责如下：

- `ClassSpaceHub`：加载首屏概览、管理三个分区和错误边界。
- `ClassSpaceSectionNav`：桌面左侧目录与平板/手机锚点条。
- `GroupChatStage`：聊天列表、历史加载、新消息提示和输入区编排。
- `GroupChatMessage`：单条消息、引用、回应和撤回状态。
- `GroupChatComposer`：文本、Emoji、引用目标、发送与重试。
- `GroupChatMineDrawer`：仅展示当前账号的历史投稿、隐藏记录和撤回记录。
- `ClassSpaceAlbumRail`：轻量相册摘要和横向滚动。
- `ClassSpaceTimelineRail`：桌面与手机统一的横向时间刻度。

`ClassSpaceHub` 不持有输入文本、表情菜单或私聊状态。群聊内部状态不应导致相册、时间轴或整页入场动画重新执行。

## 7. 公共群聊领域设计

### 7.1 复用现有公共留言表

保留 `public_messages` 作为底层消息表，避免复制历史内容和破坏现有审核记录。新增字段：

```sql
ALTER TABLE public_messages ADD COLUMN reply_to_id TEXT;
ALTER TABLE public_messages ADD COLUMN client_nonce TEXT;
ALTER TABLE public_messages ADD COLUMN recalled_by_type TEXT;
ALTER TABLE public_messages ADD COLUMN recalled_at TEXT;
ALTER TABLE public_messages ADD COLUMN moderation_reason TEXT;
```

新增幂等索引：

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_public_messages_nonce
ON public_messages(author_slug, client_nonce)
WHERE client_nonce IS NOT NULL;
```

`status` 统一解释为：

- `visible`：对全班可见。
- `hidden`：管理员临时隐藏，可恢复。
- `recalled_by_author`：作者撤回，公开区保留撤回占位。
- `recalled_by_admin`：管理员撤回，公开区保留处理占位。
- `pending`、`rejected`：仅用于历史投稿记录，不进入群聊。

迁移时把原 `approved` 更新为 `visible`；原 `hidden`、`pending` 和 `rejected` 保持原语义。新的群聊消息直接写入 `visible`。

### 7.2 表情回应

新增规范化表：

```sql
CREATE TABLE group_chat_reactions (
  message_id TEXT NOT NULL,
  reactor_slug TEXT NOT NULL,
  reaction TEXT NOT NULL,
  created_at TEXT NOT NULL,
  PRIMARY KEY (message_id, reactor_slug),
  FOREIGN KEY (message_id) REFERENCES public_messages(id) ON DELETE CASCADE,
  FOREIGN KEY (reactor_slug) REFERENCES students(slug) ON DELETE CASCADE
);
```

每位同学对一条消息最多保留一种回应。点击当前回应会取消，点击另一种会替换。第一阶段允许 `❤️`、`👍`、`😂`、`🎉`。

旧 `reactions` JSON 无法反推出具体用户，因此作为只读历史基数保留。接口返回的计数为“历史基数 + 新表聚合”，`myReaction` 只来自新表。

### 7.3 禁言记录

新增表：

```sql
CREATE TABLE group_chat_mutes (
  student_slug TEXT PRIMARY KEY,
  muted_until TEXT,
  reason TEXT NOT NULL,
  created_by TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (student_slug) REFERENCES students(slug) ON DELETE CASCADE
);
```

存在记录且 `muted_until` 为空表示无限期禁言；`muted_until` 晚于当前时间表示限时禁言。解除禁言删除记录。禁言只阻止新消息，不自动删除历史消息。

### 7.4 消息行为

- 文本去除首尾空白，长度限制为 1-500 个字符；后端始终按纯文本返回，前端不使用 `v-html`。
- 发送请求带 `clientNonce`。同一账号重复提交相同 nonce 时返回已有消息，不重复写入。
- 引用只允许指向存在的群聊消息，保存 `reply_to_id`。
- 引用目标被隐藏或撤回时，引用区显示“原消息不可用”，不泄漏原文。
- 作者可在发送后 2 分钟内撤回自己的可见消息；撤回不可恢复。
- 管理员可随时隐藏、恢复隐藏或撤回消息。隐藏可恢复，管理员撤回不可恢复。
- 管理员隐藏、撤回和禁言都必须填写原因，并创建通知。
- 消息发送采用每账号 30 秒最多 6 条、1 小时最多 60 条的基础限流；超限返回 429 和可重试时间。
- 群聊按 `created_at, id` 排序，不再使用旧留言的 `pinned`、`featured` 和 `card_style` 参与展示排序或视觉样式。
- 回应新增、替换和取消时必须同步更新父消息的 `updated_at`，使增量接口能够返回变化。
- 群聊标题栏提供“我的记录”入口，仅展示当前账号的历史 `pending/rejected` 投稿、被隐藏消息和本人撤回消息。

### 7.5 群聊接口

```text
GET    /api/group-chat/messages?before=<cursor>&limit=30
GET    /api/group-chat/mine?before=<cursor>&limit=30
GET    /api/group-chat/sync?cursor=<cursor>
POST   /api/group-chat/messages
DELETE /api/group-chat/messages/:id
PUT    /api/group-chat/messages/:id/reaction

PUT    /api/admin/group-chat/messages/:id/hide
POST   /api/admin/group-chat/messages/:id/recall
PUT    /api/admin/group-chat/mutes/:slug
DELETE /api/admin/group-chat/mutes/:slug
```

所有公开群聊接口都要求有效 `X-Classmate-Token`；管理员接口继续要求后台 JWT 和有效管理员会话。

创建消息请求：

```json
{
  "content": "周末把运动会的视频也传到相册吧。",
  "replyToId": "pm_123",
  "clientNonce": "3f7c..."
}
```

消息响应包含：

```json
{
  "id": "pm_456",
  "author": { "slug": "lisi", "name": "李四", "avatarUrl": null },
  "content": "周末把运动会的视频也传到相册吧。",
  "status": "visible",
  "replyTo": { "id": "pm_123", "authorName": "班长", "preview": "毕业照已经上传" },
  "reactionCounts": { "👍": 3 },
  "myReaction": null,
  "canRecall": true,
  "createdAt": "2026-07-10T13:30:00.000Z",
  "updatedAt": "2026-07-10T13:30:00.000Z"
}
```

游标必须是服务端生成的不透明字符串，内部至少包含更新时间和 ID，避免同一时间戳下漏消息。客户端不能自行拼接 SQL 时间条件。

### 7.6 首屏和增量同步

`GET /api/class-space/overview` 改为同学会话接口，并返回：

- 最新 30 条群聊消息和首个 `chatCursor`。
- 最多 4 个轻量相册摘要。
- 最多 8 个横向时间轴节点。
- 各分区总数。

进入班级空间只发起一次概览请求。首个群聊增量请求在 5 秒后开始，避免概览后立即重复拉取。

概览中的 `myReaction`、`canRecall` 和禁言状态与当前账号有关，因此响应必须使用 `Cache-Control: private, no-store`，不得继续使用公共 CDN 缓存。最新 30 条消息由数据库倒序截取后，在响应中按 `created_at, id` 升序返回，便于聊天窗口直接渲染。

群聊增量规则：

- 页面可见且群聊已挂载时每 5 秒请求一次。
- 页面隐藏、离线、Astro 切页或组件卸载时暂停并取消在途请求。
- 恢复可见或网络恢复时立即同步一次。
- 按消息 ID 合并，新增、回应变化、隐藏和撤回都能覆盖本地旧状态。
- 同一时刻最多一个同步请求，不允许请求重叠。

## 8. 群聊交互设计

### 8.1 消息流

- 自己的消息靠右，其他同学靠左；头像、姓名和时间保持稳定位置。
- 连续 5 分钟内同一作者的消息可弱化重复头像，但每条消息仍有可访问作者信息。
- 引用回复显示一行作者和最多 60 字预览。
- 撤回消息显示低对比度占位，不保留回应按钮。
- 管理员隐藏的消息对普通用户完全移出消息流；作者在个人记录中可看到“已隐藏”和原因。
- “我的记录”使用侧边抽屉或手机全屏面板，不占用主消息流；历史待审核和未通过投稿只在这里及后台出现。

### 8.2 滚动策略

- 初次进入默认定位到最新消息。
- 用户距离底部不超过 80px 时，新消息到达后自动滚动到底部。
- 用户正在阅读历史时不强制滚动，显示固定尺寸的“有 N 条新消息”按钮。
- 点击按钮滚动到最新消息并清零计数。
- 向上接近顶部时按游标加载更早消息，加载完成后维持当前视觉锚点，不让内容跳动。

### 8.3 发送状态

- 提交后立即插入 `sending` 临时消息。
- 成功后用服务端消息替换临时消息。
- 失败后保留文本并显示 `failed` 状态与重试按钮。
- 重试沿用原 `clientNonce`，确保网络超时后不会重复发送。
- 被禁言时输入栏显示禁言原因和解除时间，并禁止提交。

## 9. 班级信箱数据模型

现有 `mail_threads`、`mail_messages` 和 `mail_recipients` 保留为只读迁移来源；新私聊不再写入这些表。

### 9.1 固定双人会话

新增表：

```sql
CREATE TABLE direct_conversations (
  id TEXT PRIMARY KEY,
  participant_a_slug TEXT NOT NULL,
  participant_b_slug TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  UNIQUE (participant_a_slug, participant_b_slug),
  CHECK (participant_a_slug < participant_b_slug),
  FOREIGN KEY (participant_a_slug) REFERENCES students(slug) ON DELETE CASCADE,
  FOREIGN KEY (participant_b_slug) REFERENCES students(slug) ON DELETE CASCADE
);
```

参与者 slug 按字典序写入 A/B，从数据库层保证同一对同学只有一个会话。

新增消息表：

```sql
CREATE TABLE direct_messages (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  sender_slug TEXT NOT NULL,
  recipient_slug TEXT NOT NULL,
  body TEXT NOT NULL,
  client_nonce TEXT NOT NULL,
  read_at TEXT,
  created_at TEXT NOT NULL,
  UNIQUE (sender_slug, client_nonce),
  FOREIGN KEY (conversation_id) REFERENCES direct_conversations(id) ON DELETE CASCADE,
  FOREIGN KEY (sender_slug) REFERENCES students(slug) ON DELETE CASCADE,
  FOREIGN KEY (recipient_slug) REFERENCES students(slug) ON DELETE CASCADE
);
```

`read_at` 只用于收件人的未读统计，不作为“已读回执”返回给发件人。私聊消息第一阶段不可编辑和撤回。

必要索引：

- 会话参与者 A、B 与 `updated_at`。
- 消息 `conversation_id, created_at, id`。
- 未读查询 `recipient_slug, read_at, created_at`。

### 9.2 旧邮件迁移

迁移必须在测试数据和生产备份上先验证：

表结构由 D1 migration 创建；旧数据归并由仓库内一次性 TypeScript 迁移脚本执行。脚本使用稳定 ID和 `INSERT OR IGNORE` 保证幂等，并在结束时输出源线程数、源消息数、新会话数、新消息数、通知数和异常记录，发布流程只有在异常记录为零且数量核对通过时才能继续。

1. 对每个学生创建的 `private` 旧线程，取创建者和收件人作为参与者。
2. 按 slug 排序，创建或复用唯一 `direct_conversations`。
3. 将该线程内的全部学生消息按时间复制到 `direct_messages`；消息接收者永远是会话中的另一位同学。
4. 同一对同学原有多个标题线程时合并到同一个会话，按 `created_at, id` 排序。
5. 保留原消息 ID；迁移消息的 `client_nonce` 固定为 `legacy:<原消息 ID>`，迁移脚本可重复执行且不得重复插入。
6. 迁移后校验每个旧学生私信线程的消息数量、参与者和时间顺序。

旧管理员和系统线程不进入私聊：

- 每个收件人对应生成一条通知。
- 标题保留为通知标题。
- 线程内历史内容按时间整理为通知正文，不能丢失学生曾经的回复。
- 原 `read_at` 映射到通知已读状态。
- 迁移通知 ID使用稳定前缀，脚本可安全重跑。

迁移成功并完成一个版本的线上观察后，旧表仍暂时保留，不立即删除。

## 10. 班级信箱接口

### 10.1 私聊接口

```text
GET  /api/direct-conversations
POST /api/direct-conversations
GET  /api/direct-conversations/:id/messages?before=<cursor>&limit=30
POST /api/direct-conversations/:id/messages
PUT  /api/direct-conversations/:id/read
```

`GET /api/direct-conversations` 返回当前账号作为 A 或 B 的全部会话，包含对方姓名、头像、最后一条消息、更新时间和未读数量。查询不能只按“收件人”过滤。

开始新会话请求不包含标题：

```json
{
  "recipientSlug": "lisi",
  "body": "周末有空一起回学校看看吗？",
  "clientNonce": "b91e..."
}
```

若会话已经存在，接口直接在已有会话中添加消息；若不存在，则在同一事务中创建会话和首条消息。不能创建空会话，也不能给自己发送消息。

读取会话和发送消息都必须验证当前账号是参与者。读取任意非参与会话统一返回 404，避免泄漏会话是否存在。

读取消息列表本身不自动标记已读。详情完成渲染后，前端调用 `PUT /api/direct-conversations/:id/read` 并提交最后一条已展示的收件消息 ID；后端只将该消息及更早的当前账号收件消息标记为已读。

### 10.2 通知接口

继续使用 `notifications` 表和现有已读接口，扩展通知类型：

- `admin_notice`
- `system_notice`
- `group_chat_hidden`
- `group_chat_recalled`
- `group_chat_muted`
- `group_chat_unmuted`
- 既有的审核结果类型

管理后台新增面向单人和全班的通知发送接口。现有 `/api/admin/mail/send` 与 `/api/admin/mail/broadcast` 在一个兼容版本内转发到通知服务，并记录弃用，不再创建新邮件线程。

### 10.3 增量同步和未读摘要

```text
GET /api/inbox/summary
GET /api/inbox/sync?cursor=<cursor>
```

`/api/inbox/summary` 返回：

```json
{
  "directUnread": 3,
  "notificationUnread": 2,
  "totalUnread": 5
}
```

`/api/inbox/sync` 返回当前账号自游标后的新私聊消息、发生变化的会话摘要、新通知和最新未读统计。游标由服务端生成，查询时刻作为下一游标边界，确保请求期间到达的消息进入下一次同步。

信箱页面可见时每 5 秒增量同步；页面隐藏或卸载时停止。导航不复用信箱的 5 秒轮询，只响应 `alumni:inbox-changed` 并保持自己的 60 秒低频摘要同步。

## 11. 班级信箱界面

### 11.1 桌面端

页面使用工作区布局，不再把完整信箱包在另一张装饰卡片中：

- 左栏宽度为 320-360px，顶部是“私聊 / 通知”分段控件和搜索。
- 私聊列表每项显示头像、姓名、最后消息、时间和稳定尺寸的未读角标。
- 通知列表显示来源印章、标题、摘要、时间和未读状态。
- 右侧为聊天或通知详情，顶部固定会话对象，中央独立滚动，底部为输入栏。
- 未选择内容时显示简洁的信纸空状态，不显示功能说明大卡片。
- 新建私聊使用同学选择面板；选中同学后打开已有会话，或在发送首条消息时创建会话。

私聊没有标题输入框。通知详情没有回复输入栏。

### 11.2 手机端

- 初始只显示私聊或通知列表。
- 点击条目进入全屏详情，顶部显示返回按钮、对象名称和必要操作。
- 当前详情 ID写入 URL 查询参数，刷新后可恢复；浏览器返回优先回到信箱列表。
- 输入栏固定在安全区上方，虚拟键盘弹出后不能遮挡最后消息。
- 列表和详情分别保存滚动位置，返回列表时恢复原位置。

### 11.3 私聊消息状态

- `sending`：本地半透明显示发送中。
- `sent`：服务端确认后正常显示。
- `failed`：保留内容并提供重试图标按钮。
- 收到新消息时，仅在用户接近底部时自动滚动；否则显示新消息按钮。
- 重试使用相同 `clientNonce`，与群聊保持一致的幂等策略。

## 12. 管理后台最小改造

### 12.1 群聊管理

- 按最新、已隐藏、已撤回筛选。
- 查看消息作者、引用关系、回应数量和处理记录。
- 执行隐藏、恢复、管理员撤回和禁言。
- 处理动作必须填写原因；禁言可选时长或无限期。
- 处理成功后同步生成站内通知。

历史 `pending` 和 `rejected` 投稿保留在“历史公共投稿”筛选中，不与实时群聊混排。

### 12.2 通知管理

原“班级邮局”管理入口改名为“通知中心”：

- 支持给单个同学发送通知。
- 支持全班广播。
- 显示发送时间、接收人数和已读数量。

## 13. 动画与加载稳定性

### 13.1 动画所有权

- 全局 reveal 只处理静态页面阅读区，不处理导航、群聊消息、信箱列表和输入栏。
- 群聊舞台首次挂载只执行一次 220ms 淡入；随后新消息仅对新增节点执行 160ms 位移和透明度过渡。
- 相册和时间轴进入视口时各执行一次，不因群聊轮询、回应或发送消息重播。
- 导航、抽屉、下拉菜单和未读角标由导航模块独占动画。
- 同一元素不得同时具有全局 reveal 和组件 transition。

### 13.2 性能规则

- 动画只使用 `transform` 和 `opacity`，不持续动画化阴影、模糊、宽高或布局位置。
- 群聊和私聊列表使用稳定的消息键，不因轮询重新创建全部 DOM。
- 初始班级空间不加载完整相册、灯箱、完整时间轴或旧公共留言组件。
- 相册封面有固定 `aspect-ratio`、显式宽高和懒加载。
- 聊天骨架高度接近最终舞台高度，加载完成后不改变页面主结构。
- `prefers-reduced-motion` 下取消滑动墨线、错落入场和平滑滚动，状态变化立即完成。

### 13.3 请求生命周期

每个轮询模块必须具备：

- 单一计时器。
- 单一 `AbortController`。
- 请求重叠保护。
- `visibilitychange` 暂停与恢复。
- `online` 恢复同步。
- Vue 卸载和 `astro:before-swap` 清理。

不得通过全局 `setInterval` 留下跨页面存活的匿名轮询器。

## 14. 错误处理

### 14.1 班级空间

- 概览失败时，群聊、相册和时间轴显示统一页面级错误与重试。
- 概览成功后，后续群聊同步失败只显示连接状态，不清空已有消息。
- 连续失败采用 5、10、20、30 秒退避；恢复后回到 5 秒。
- 历史消息加载失败保留当前位置并允许重试。
- 消息发送失败保留内容和引用目标。

### 14.2 信箱

- 会话列表失败、当前会话历史失败和增量同步失败分别处理。
- 详情失败不能清空左侧列表。
- 发送失败保留临时消息并允许重试。
- Session 失效统一清理本地会话并回到登录页；403 权限错误不能假装成网络错误。

### 14.3 管理操作

- 隐藏、撤回和禁言使用服务端最终状态；失败时恢复按钮状态。
- 重复管理员操作应幂等，不能生成重复通知。
- 数据迁移失败必须整体回滚或停止发布，不允许部分用户进入新旧混合信箱。

## 15. 安全、隐私与可访问性

- 群聊、私聊、通知和未读摘要都要求有效同学 Session。
- 私聊所有查询必须在 SQL 层限制当前账号是参与者，不能只依靠前端隐藏。
- 管理操作同时验证 JWT 和 `admin_sessions`。
- 文本内容按纯文本渲染，禁止将聊天内容插入 `innerHTML`。
- 对消息、回应和同步接口增加请求体大小限制与账号级限流。
- 管理通知原因不能包含其他同学的私密数据。
- 图标按钮使用 Lucide 图标，并提供中文 `aria-label` 和工具提示。
- 消息列表使用 `role="log"` 和适度的 `aria-live="polite"`，历史批量加载不得一次朗读全部消息。
- 手机触控目标不小于 44px；键盘可完成菜单、筛选、发送、回应和返回操作。
- `Escape` 关闭 Emoji 菜单、更多菜单、移动抽屉和收件人选择面板。

## 16. 兼容与下线策略
