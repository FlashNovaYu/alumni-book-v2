# 同学录多管理员后台升级设计

## 目标与边界

将当前“单一管理密码 + 单一 `admin` 角色”的后台升级为可追责的多管理员工作台。主管理员拥有最高权限；次级管理员只能管理被授予的业务模块，不能查看同学名单或同学档案。升级后需要在桌面和手机上完整完成内容审核、内容运营和通知发布。

本次包含：

- 独立管理员账号与绑定同学账号的次级管理员入口；
- 预设角色加单账号权限增减的 RBAC；
- 对现有审核、通知、相册、时间轴等模块的服务端权限保护；
- 管理员账号、权限与审计日志的主管理员工具；
- 按权限动态显示的桌面／手机管理工作台；
- 旧单密码主管理员的无中断迁移。

本次不包含：

- 改变同学账号登录或同学资料自助编辑规则；
- 向次级管理员暴露同学名单、同学档案或相关导出能力；
- 改变公共站点的可见性规则、同学数据模型或已有内容业务规则；
- 主管理员对次级管理员操作的二次审批。次级管理员在获授权范围内的操作即时生效。

## 已确认的产品决策

| 主题 | 决定 |
| --- | --- |
| 授权模型 | 预设角色与单账号权限覆盖并存。角色提供默认权限，主管理员可对单个账号增减。 |
| 主管理员 | 只能是独立管理员账号，拥有全部权限。 |
| 次级管理员 | 可是独立管理员账号，也可绑定一个现有同学账号；绑定账号不能成为主管理员。 |
| 默认业务范围 | 次级管理员可获授所有非核心业务模块权限；核心权限为同学档案编辑、管理账号、权限配置、站点核心设置。 |
| 同学资料 | 次级管理员不能查看名单或档案详情，也不能导出。 |
| 内容决策 | 次级管理员的获授权审核、撤回、隐藏、通知发布等操作即时生效，不需要二次批准。 |
| 审计 | 所有管理写操作生成可检索日志，含操作者、时间、对象、原因与变更摘要。 |
| 账号停用 | 不直接删除账号。停用立即撤销会话和管理入口，账号与日志保留。 |
| 旧管理员迁移 | 保留现有密码作为一次性迁移凭据；首次登录后补全用户名、显示名并设置、确认新密码。 |
| 设备 | 桌面与手机均为正式管理入口；手机可以完成审核、处理和发布，不只是只读看板。 |

## 方案选择

采用数据库驱动的 RBAC，而非硬编码角色或把权限写进 JWT。

原因：用户需要角色模板之外的逐账号增减权限；权限变更和账号停用必须立即生效。令牌只标识会话与账号，不作为权限真相。Worker 在每次受保护请求中读取活跃账号及其最终权限；前端只据此展示可见入口，不能替代服务端鉴权。

## 数据模型与迁移

现有 `admin_sessions` 只保存 token 和过期时间，`/api/auth/login` 从 `site_config.admin_password` 验证单个密码并签发 `{ role: 'admin' }`。升级新增以下模型，并将会话关联到具体管理员。

### 管理员账号

`admin_accounts`：

- `id`：稳定主键；
- `account_type`：`standalone` 或 `classmate_linked`；
- `username`：独立账号必填且唯一；绑定同学账号为空；
- `display_name`：后台显示名称；
- `student_slug`：绑定同学账号时必填且唯一，独立账号为空；
- `password_hash`：独立账号使用 PBKDF2；绑定同学账号为空；
- `must_change_password`：独立账号初始密码或重置密码时为真；
- `status`：`active` 或 `disabled`；
- `is_owner`：仅一个活跃独立账号可为真；系统不允许停用、降级或移除唯一主管理员，避免失去最高权限入口；
- `last_login_at`、`created_at`、`updated_at`。

绑定约束保证一个同学账号最多绑定一个管理员身份；主管理员不得绑定同学账号。密码不存明文。

### 角色和权限

`admin_roles` 保存角色 ID、名称、系统标记和描述；`admin_role_permissions` 保存角色默认权限；`admin_account_permissions` 保存单个账号的 `allow` / `deny` 覆盖；`admin_accounts.role_id` 指向一个角色。每个账号仅有一个基础角色，避免多个角色叠加后难以解释最终权限。

最终权限的计算顺序为：主管理员拥有全部权限；否则先读基础角色权限，再应用账号的 `deny` 和 `allow` 覆盖。每次账号编辑页保存前展示“最终拥有的权限和模块”。

首版权限键按业务能力分组，而不是按页面名称：

| 分组 | 典型权限 |
| --- | --- |
| 审核治理 | 查看审核队列、审核内容、隐藏／恢复、撤回、禁言／解禁 |
| 内容运营 | 发布通知、管理相册、管理时间轴、管理其他现有公共内容 |
| 同学档案 | 查看／编辑同学名单和资料；仅主管理员 |
| 后台治理 | 管理管理员账号、角色／权限、站点核心设置、查看审计日志；仅主管理员 |

预设角色为：

- **主管理员**：全部权限，不能是绑定同学账号；
- **内容管理员**：全部非核心业务权限；
- **内容审核员**：群聊、留言与其他内容的审核治理权限；
- **运营管理员**：通知、相册、时间轴及其他内容运营权限。

### 会话与审计

`admin_sessions` 扩展为关联 `admin_account_id`，并保留独立的到期时间、创建时间和撤销信息。每次管理 API 请求都验证会话未过期、账号未停用，并重新计算当前权限。账号停用、改权、角色变更、撤销会话无需等待 JWT 过期即可生效。

`admin_audit_logs` 记录：`id`、`admin_account_id`、`action`、`resource_type`、`resource_id`、`reason`、`before_summary`、`after_summary`、`metadata`、`created_at`。日志不可由次级管理员删除或修改；主管理员可筛选和查看，但不提供清除功能。

业务修改与对应审计日志必须通过同一次 D1 batch / 事务提交：如果审计写入失败，业务修改也失败，避免出现无法追责的成功操作。

### 首次迁移

迁移不触碰 `students`、内容表或同学会话。

1. 添加新表和会话关联字段，保留 `site_config.admin_password` 作为一次性迁移凭据。
2. 当没有主管理员账号时，现有仅密码登录进入“主管理员初始化”流程，而非直接进入业务后台。
3. 使用旧密码验证成功后，签发短时、仅能完成初始化的令牌。
4. 初始化页要求用户名、显示名和新密码二次确认；成功后创建唯一主管理员账号、写入 PBKDF2 密码哈希、撤销旧管理会话并使旧密码登录入口失效。
5. 后续登录一律通过管理员账号；旧配置不再用于认证。

## 登录与鉴权流程

### 独立管理员

用户输入用户名和密码。Worker 验证账号状态、密码和是否要求改密，创建关联账号的管理会话。初始密码或管理员重置后的密码只能进入改密流程；改密成功前不得调用业务接口。

### 绑定同学账号的次级管理员

同学先按现有方式登录。前端只在当前同学绑定了活跃次级管理员且拥有至少一个业务权限时显示“进入管理后台”入口。点击后，Worker 使用 `X-Classmate-Token` 验证同学会话、查找绑定管理员账号和其状态，创建独立的短时管理会话。该会话不继承同学资料权限，且无法调用主管理员专属接口。

### 中间件规则

现有按 HTTP 方法给整个管理路径套用的单一 JWT 校验替换为：

1. `requireAdminSession`：会话、账号状态和初始化／改密状态；
2. `requirePermission(permission)`：当前账号的最终权限；
3. `requireOwner`：主管理员专属路径。

每个写接口按能力映射到一项权限；查看管理模块的数据同样要求最小查看权限。前端路由守卫启动时请求 `/api/auth/me`，得到当前账号资料、角色和最终权限。前端导航、工作台、快捷操作和页面内按钮都以这些权限为准，但服务端始终重复校验。

## 管理后台体验

### 桌面工作台与导航

登录后的首屏是已确认的混合工作台：

- 优先区：我的待办，显示群聊审核、公共留言、通知草稿等数量和一键处理入口；
- 辅助区：近 7 日互动、待发布内容等轻量运营信息；
- 快捷操作：仅显示当前账号有权执行的发布、审核、归档操作；
- 侧栏分组为“工作台”“内容治理”“内容运营”；
- “同学档案”“账号与权限”“站点设置”属于主管理员分组，次级管理员的导航、路由和请求均不包含这些内容。

审核中心统一承载现有的个人留言、公共群聊和历史公共投稿治理。通知中心、相册管理和时光轴保留现有业务能力，逐页加上权限守卫。初版不做与权限升级无关的功能重构。

### 手机管理端

手机使用底部主导航：工作台、审核、运营、更多。工作台首屏展示待办，审核详情可完成完整处置，通知发布页可完成完整发布。复杂的账号和权限批量管理只在主管理员的“更多”中打开独立页面；桌面仍是高密度批量操作的主要场景。

导航由 `/api/auth/me` 的权限结果驱动，无权模块不渲染也不预取数据。所有交互保留可访问的标签、焦点顺序和明确的加载／失败状态。

### 主管理员专属页面

**账号与权限**页面展示管理员的显示名、登录类型、角色、状态、最后活动时间和管理入口。新建账号按以下流程完成：

1. 选择独立账号或绑定已有同学账号；
2. 为独立账号填写用户名、显示名和初始密码，要求首次改密；绑定账号选择同学并填写显示名；
3. 选择预设角色，显示最终权限；
4. 用开关增加或拒绝特定业务权限后保存。

停用按钮需确认，成功后立即撤销该账号所有管理会话、隐藏其绑定同学账号的后台入口。唯一主管理员不可停用、降级或删除。主管理员可重置独立账号密码并撤销其会话。账号列表不提供硬删除。

**审计日志**页面只对主管理员开放，支持操作者、模块／动作、日期和对象筛选；详情显示理由和变更摘要，必要时显示前后字段差异，不直接暴露无关同学资料。

## 审核与反馈行为

审核员进入待办后可查看完整内容上下文、作者、当前状态和历史处置。根据权限显示通过、隐藏／恢复、撤回、置顶、禁言等操作；破坏性操作要求填写非空原因。操作成功后待办计数和页面状态更新，且审计记录与业务修改同时提交。

统一错误处理：

| 情况 | 行为 |
| --- | --- |
| 401 会话无效 | 清理管理会话并回到相应登录入口。 |
| 403 权限已收回 | 重新拉取当前账号，提示权限变化，并导航至仍有权的工作台。 |
| 409 内容已被处理 | 不覆盖他人决定；刷新内容状态，显示最新处理时间和操作者。 |
| 验证失败 | 就地显示字段错误，保留输入；不创建审计记录。 |
| 网络或服务端失败 | 保留处理理由与表单，不显示成功状态；用户可主动重试。 |

## 验证策略

Worker 测试覆盖：

- 旧密码首次迁移、初始化完成后旧入口失效；
- 独立管理员登录、强制首次改密、重置密码、会话撤销；
- 绑定同学账号的入口只为活跃、授权的次级管理员创建会话；
- 主管理员不能绑定同学账号，绑定账号不能被提升为主管理员；
- 每个权限允许和拒绝对应接口；收回权限或停用账号后，已有会话立即得到 403 / 401；
- 审核业务写入和审计日志一起成功或一起失败；
- 次级管理员不能访问同学目录、档案、账号、权限、设置和审计接口。

管理后台的类型检查和构建必须通过。Playwright 覆盖桌面、平板和手机：路由与导航可见性、混合工作台待办、审核处置、权限即时收回、手机底部导航、无横向溢出、键盘可达性。测试使用独立管理员与绑定同学账号两种身份。

## 实施范围与顺序

1. 建立 D1 迁移、权限库、认证会话和 Worker 中间件，并为安全边界补充测试；
2. 迁移现有主管理员登录，新增账号、权限、审计 API；
3. 将现有业务路由按权限收紧，确保次级管理员无同学数据读取路径；
4. 升级管理客户端、路由守卫和动态导航，加入混合工作台及手机导航；
5. 实现账号与权限、审计日志、审核中心的管理界面；
6. 跑 Worker、管理后台构建和跨视口端到端验收。

每一步保持最小改动：复用现有 Hono 路由、D1、PBKDF2、`adminFetch` 和 Vue Router 模式，仅替换单管理员认证与缺失的权限治理层。
