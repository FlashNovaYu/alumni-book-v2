# 移动端同学档案卡连续同步过渡设计（第二版）

## 背景与根因

现有手机端方案把进入和返回总时长延长到 `1.05s`，但进入关键帧在前 24% 只把卡片四边扩出 `2vmax`，随后配合强烈的 ease-out 曲线集中铺满屏幕。390×844 视口逐帧采样显示：

- 进入约 335–436ms 时，背景覆盖范围从卡片附近突然跳到接近半屏，形成“先停住、再猛地铺开”的观感。
- 头像与姓名约 377ms 开始移动；约 460ms 时身份元素已经越过仍停留在卡片附近的背景边界。
- 返回动画整体比进入同步，但开头从 overscan 到屏幕边缘收缩过快。
- `max-width: 768px` 无法覆盖部分横屏手机。

因此问题不是总时长不足，而是背景边界与共享身份元素使用了不协调的进度曲线。

## 目标

- 手机端进入时，背景从第一段就持续向外扩散，不再出现长时间近乎静止后的突跳。
- 头像和姓名在扩散早期开始移动，但在任一可见采样点都不得越过背景扩散边界。
- 返回时使用相反的连续节奏：身份元素先向卡片收拢，背景边界始终包住身份元素，最后同时落位。
- 覆盖常见手机竖屏和横屏，同时保持桌面端现有效果不变。
- 保留 `prefers-reduced-motion` 的无动画降级。

## 方案

继续使用现有 Astro View Transitions、卡片几何 CSS 变量和共享元素命名，不新增 JavaScript 动画控制器。

### 1. 适用范围

手机规则使用以下等价条件：

- 所有 `max-width: 768px` 的窄视口；
- `pointer: coarse` 且 `max-width: 900px` 的触摸横屏设备。

桌面鼠标设备和宽屏继续使用当前桌面参数。

### 2. 进入节奏

- 背景扩散总时长保持约 `1.05s`。
- 删除“24% 仅扩出 2vmax、88% 才铺满”的停顿式关键帧。
- 卡片矩形到全屏边缘采用单段连续插值，并在最后约 8% 只完成 overscan，避免边缘露缝。
- 使用温和的 ease-out 曲线，使背景边界在中前段略领先身份元素，但不产生突跳。
- 头像和姓名延迟约 `0.08s` 开始，移动约 `0.92s`，在背景完成覆盖时落位。
- 卡片辅助信息在约 `0.52–0.56s` 内向中心坍缩并淡出，不在背景已经展开后继续残留。

### 3. 返回节奏

- 返回总时长约 `0.95–0.98s`，略短于进入，避免返回显得拖沓。
- 身份元素比背景收缩更早开始，移动约 `0.86s`；背景边界始终包住正在回到卡片的头像和姓名。
- 背景从全屏连续收回卡片矩形，不再在开头快速跳过 overscan 区间。
- 返回落位后继续复用已实现的原滚动位置恢复，不增加新的路由状态。

## 性能约束

- 只动画 View Transition 快照的 `clip-path`、`transform` 和 `opacity`。
- 不在逐帧回调中读取布局或修改 DOM。
- 不引入 GSAP、Canvas 或新的运行时依赖。
- 在 4 倍 CPU 降速的 Chromium 手机视口下，不应因本次修改新增持续长任务。

## 测试与验收

### 静态合同

- 锁定手机媒体条件、进入/返回时长、身份元素延迟和持续时间。
- 禁止重新引入“前段只扩 2vmax、接近结束才铺满”的停顿式关键帧。
- 桌面端参数和 reduced-motion 规则保持不变。

### 浏览器行为

- 在 390×844 手机视口验证进入与返回完整完成。
- 采样背景 clip 边界与头像/姓名 View Transition group 的几何位置：进入和返回过程中，身份元素边界不得超出背景可见区域。
- 背景覆盖面积必须单调连续变化，不出现单个短时间区间内的集中突跳。
- 快速点击、返回滚动位置、陈旧学生状态和减少动态偏好用例继续通过。

### 发布验收

- 使用干净提交构建阿里云自托管产物。
- GitHub Verify 通过后再发布。
- 线上静态 `/release.json` 与 `/api/health` SHA 一致，readiness 为 true。
- 实际手机视口确认进入、返回、横屏规则均来自已发布 CSS。

## 非目标

- 不修改桌面端档案卡动画风格。
- 不重写 View Transition 状态机。
- 不为不同手机型号维护独立时长表。
- 不修改学生数据、API、数据库或认证逻辑。
